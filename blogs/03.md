# Why Agents Need a Hub

> *"The knowledge of the circumstances of which we must make use never exists in concentrated or integrated form, but solely as the dispersed bits of incomplete and frequently contradictory knowledge which all the separate individuals possess."*
>
> — F.A. Hayek, *The Use of Knowledge in Society* (1945)

The [previous post](02.md) established that a transmission between agents should occur if and only if both sides come out ahead, and measured the quality of an agent network in terms of Coverage, Precision, and Cost. The target is now set. What remains is the architecture that reaches it. This post formulates the visibility problem and argues that a hub-and-spoke structure is the appropriate solution at scale.

## Agent Mutual Invisibility

Hayek's starting observation was deceptively simple. In any real economy, each participant acts from a particular situation shaped by a large number of local details, including resources, preferences, constraints, and intentions that vary across individuals and shift continuously. The behavior that results is complex enough that no single mind can hold a summary of it, and the information about supply and demand therefore exists only as dispersed fragments, each held by a separate participant.

Because this information is dispersed, participants need a place to declare it. Each participant must be able to signal what it holds and what it seeks. We call the infrastructure that makes this possible the discovery layer. Human societies have spent centuries constructing theirs. Physical markets let buyers and sellers converge in the same location. Platforms like LinkedIn do the same for labor markets, bringing employers and candidates into a shared digital space where opportunities can be found. Across these and many other domains, human society has arrived at a distributed consensus about where particular classes of supply and demand are meant to meet.

Agents face the same structural situation. In practice, each agent is generally deployed to serve a particular principal, either a person or an organization, and is configured to serve that principal's specific purposes. Its tools, memories, preferences, and operating environment are shaped by those needs, and those needs differ enough across principals that no centralized deployment can accommodate all of them. The consequence is that agents are deployed in a distributed fashion, running on personal laptops, local servers, and cloud infrastructure alike. This is the configuration the agent ecosystem has today, and there is no reason to expect it to change.

Distributed deployment produces the same consequence for agents that dispersed knowledge produces for people. Each agent is invisible to every other. Nothing in the current infrastructure tells one agent that another exists, what it holds, or what it needs. To coordinate, agents require the same precondition human markets evolved to provide, a discovery layer where supply and demand become mutually visible. That layer does not yet exist for agents. We call its absence **agent mutual invisibility**.

The result is a stark default. Two agents working the same domain can be running at the very same moment, each holding precisely what the other needs, and never find each other. Nothing connects them because nothing has been built to. In the terms of the previous post, a transmission satisfies $S > 0$ and $D > 0$ only once both sides are known to the network. Today, neither is. We call this the **visibility problem**, and it is the precondition every architectural choice in this post has to answer to.

## The Status Quo

Without a dedicated structure, agents are left with three improvisations: receiving, publishing, and searching. None of them solve the visibility problem.

### Receiving: crawl the world yourself

Suppose your agent watches career opportunities on your behalf. It must poll LinkedIn, Glassdoor, Indeed, individual company pages — hundreds of sources daily in case something new appears. No channel exists where hiring agents announce openings *to* candidate agents. The opportunity exists somewhere, the agent exists to receive it, and no channel connects them. The same pattern repeats for any standing watch task — research collaborators, suppliers, domain experts.

Each watching agent pays this cost in isolation. When $n$ agents in the same domain each crawl their own sources, the network performs $O(n)$ redundant work per source and $O(n^2)$ in total — every agent paying full freight to discover the same surface.

### Publishing: blast it everywhere

The employer's side mirrors the problem. Its agent can filter candidates by keywords, but keyword matches return hundreds of false positives — wrong seniority in practice, wrong culture fit, wrong trajectory. The agent ends up pushing the opening to every plausible match. Every candidate agent receives a flood of openings that look relevant but aren't. Each mismatch costs tokens to receive, parse, and discard. In the framework of the [previous post](02.md), $D < 0$: the receiver pays real cost for information of no value.

### Searching: ask for what you already know to ask for

The career-watching agent can search for *"senior backend roles at Series B startups in Berlin"* because it already knows that is what it wants. It cannot search for the stealth-mode company that raised overnight, the regulatory change in an adjacent jurisdiction that will reshape hiring in six months, or the researcher whose work intersects its domain in ways neither side has articulated. Search retrieves what was asked for; the matches that move an agent's decisions the most are the ones it could not have asked for.

Improving search does not fix this. A better retrieval engine still answers queries; the gap is that the most valuable signals don't enter the network as answers to queries at all.

## Why a Hub

Crawling, blasting, and searching share a single root failure. No participant sees the full supply-demand surface. Whatever structure replaces them has to put that surface somewhere — and the structure that wins is determined by who can see it most efficiently.

The obvious counter-proposal is to keep things decentralized — every agent maintaining its own recommendation model, training on its own signal, serving its own matches. A deeper variant pulls rather than pushes: a network of crawlers that fetch each agent's internal state to infer what it has and what it wants. Both fail, for two reasons that compound.

First, supply cannot be inferred from the outside. Only the sender knows what it is willing to share, on what terms, and under what embargoes; a crawler can read what an agent has written but cannot read what an agent has chosen not to write.

Second, reproducing matching at every node returns the network to its starting point. Each agent runs its own crawler, builds its own model, and reconstructs the same $O(n^2)$ work the discovery layer was supposed to remove. A hub collapses this to $O(n)$: each agent declares once, the hub matches once.

The pattern is not unique to agents. YouTube, Spotify, and TikTok exist as centralized recommenders rather than as personal models on each laptop. The same two forces apply here.

**Specialization needs scale.** A recommendation system has a fixed cost — engineering, ranking models, freshness detection, per-domain tuning — that pays back only when amortized over enough traffic. But cost is the smaller half of the story. A private recommender running on one agent's signal never sees enough of the surface to learn the patterns that matter; the matches worth making live in the cross-traffic between domains, and only become visible to a system that watches every spoke at once. This is Stigler's extension of Smith: the division of labor is limited by the extent of the market, so the depth of any specialized function is set by the scale that sustains it. A fragmented landscape of private recommenders never reaches the scale at which specialization pays. Not because each node is too poor, but because each node is too blind.

**Matching needs a thick, ordered, safe market.** A working matching system needs three things at once: enough participants that a match is likely to exist (thickness), enough throughput that candidates can be evaluated before they go stale (low congestion), and enough oversight that participants can reveal what they actually want without being punished for it (safety). These are Alvin Roth's conditions for a functioning matching market. A decentralized arrangement reaches none of them cleanly: thickness fragments across nodes, congestion control requires a global view of the queue, and safety requires that misbehavior on one edge be visible at every other. A hub satisfies all three because it is the single place where supply, demand, and behavior are jointly observable.

The result is a **hub-and-spoke** topology. Every agent (spoke) connects to the hub, the hub sees the complete picture, and matching happens at the center. From this position the hub does two things at once. It makes matches that no spoke could have found independently — the career-watching agent and the hiring agent from earlier no longer brute-force discovery. And it governs the network, because every transmission passes through it. The first job is the subject of this post; the second is the subject of the next one.

## Three Core Primitives

A hub-and-spoke network needs three core primitives. **Publish**, by which a spoke expresses supply. **Profile**, by which a spoke exposes demand. **Push**, by which the hub delivers a match.

### Publish

**Publish** is the primitive by which an agent declares supply to the hub. It is a statement that a piece of information exists, what it contains, and under what conditions the sender is willing to distribute it — not a broadcast to anyone in particular. The previous section explained why declaration has to be active rather than scraped; what remains is what the act of publishing looks like.

A news agent, a legal agent, a research agent each publish once. The hub routes to exactly the right receivers across the network, without the sender needing to know who they are. Because supply changes over time, Publish must also be retractable — outdated or erroneous supply needs a corresponding **Delete**.

One constraint governs what is worth publishing. The information must have **decision-changing potential**. Content that produces no decision-relevant value still imposes token cost on receivers, and therefore harms the network.

### Profile

**Profile** is the primitive by which an agent exposes demand. Without it, the hub can receive supply but has no basis on which to route it.

A query is the obvious way to express demand, but it is the wrong shape. The highest-value matches, as the previous section argued, are precisely the ones the agent itself didn't know to ask for. A query returns what was asked for; it cannot reach into what an agent didn't know to look for.

What the hub needs instead is a standing description of what each agent is, what it cares about, and what would shift its decisions — rich enough that the hub can recognize relevant supply on the agent's behalf before the agent could have asked. That standing description is a Profile. The richer the Profile, the larger the decision surface over which the hub can spot relevance the agent itself could not have articulated in advance.

### Push

Once the hub matches supply to a receiver, it pushes immediately. Two reasons make this the only correct delivery model.

**Information decays.** A security vulnerability, a market-moving event, a regulatory change all lose value by the hour. Any model that waits for the agent to come looking destroys utility during the gap.

**Agents have no attention bottleneck.** For humans, push is a real tradeoff — it spares the polling cost but imposes an attention tax. For agents, that counter-cost collapses. Agents pay per token, not per attention slot, so they cannot be overwhelmed, only overbilled. Push already minimizes the only cost they pay. If a transmission has positive net utility, the receiver is strictly better off getting it the moment it lands.

The full picture only comes into focus when we look at the cumulative effect of Push, applied across every match the hub makes for a single receiver. What arrives is not a discrete event but a steady stream — continuous, personalized, and shaped by who the agent is and what it needs. No two agents see the same one. This stream is novel enough to deserve a name; we call it the agent's **flux**.

A flux changes what an agent can be. Without one, an agent is reactive — it acts when prompted, searches when asked, and knows only what it is told to find. With matched information arriving continuously, an agent becomes proactive: the litigation agent surfaces a new ruling the moment it is published, before its user has thought to ask whether one exists.

## From Match to Channel

A match is rarely the end of the exchange. Once the hub connects two agents, the work continues — terms, follow-ups, clarifications — none of which fits a broadcast shape. The hub handles this by issuing the two sides a persistent reference to each other after the match: a verified handle each can use to reach the other directly. A private spoke-to-spoke channel exists where, a moment earlier, neither side knew the other was on the network.

A first message is only worth opening because the hub stands behind both ends. It verified identity, it judged the match, and either side can break the channel if it stops being useful — and that exit is real only if misbehavior elsewhere on the network is visible here. Identity that cannot be spoofed, reputation that tracks behavior, accountability when something is misrepresented: none of the primitives in this post are safe at scale without them. That trust layer is the subject of the next post.
